# ナレッジノート生成・修正

## モード判定

- 既存ファイルへの修正と判断できる場合 → **修正モード**（「修正して」、「整理して」、既存ファイルの指定がある等）
- 新規ファイルの作成と判断できる場合 → **作成モード**（「メモして」、「記録して」等）

## 修正モード

対象は任意のmarkdownファイル（`./note/` 配下に限らない）。

### 手順

1. 対象ファイルを特定する（コンテキストにあるファイル、またはユーザーが指定したファイルを読み込む）
2. テンプレートとフォーマット、ファイル命名のルールに照らして不整合を確認する
3. 修正内容をユーザーに提示し、承認後に適用する

### 原則

- 内容（事実・記述）は変更しない。構成とフォーマットのみ修正する
- セクションの追加・並べ替えはテンプレートの構成に合わせる
- 既存の記述を削除しない。テンプレートに該当セクションがない場合は末尾に残す

## 作成モード

### 手順

1. 会話内容から記録すべきトピックを特定し、カテゴリを判定する
2. 独立したトピックが複数あればファイルを分割する
3. `./note/` の既存ファイルを確認し、トピックが一致するファイルがあれば
   追記か新規作成かをユーザーに確認する。追記の場合は修正モードに切り替える
4. テンプレートをもとにmarkdownファイルを作成・保存する
5. 保存先のフルパスを伝える

### 出力先

- `./note/`（作業ディレクトリ直下）に保存する。なければ作成する
- サブディレクトリは自動作成しない。提案のみ行う（例: `./note/Rust/`）

## ファイル命名

- 後から検索できる具体的なトピック名を使う（形式: `トピック名.md`）
- 良い例: `PatternLayoutのスタックトレース表示.md`、
  `Rustのライフタイムとnon-lexical-lifetimes.md`
- 悪い例: `Pythonメモ.md`（広すぎる）、`調査結果.md`（内容不明）、
  `2026-03-29.md`（日付だけ）

## テンプレート

内容に最も適したカテゴリを選択する。
セクションの追加・省略・並べ替えは自由に行う。
共通構成: `# タイトル` → `> 1〜2文の要約` → `## セクション`

- **技術調査**: ポイント → Q&A → 参考
- **トラブルシューティング**: 症状 → 原因 → 解決策 → 教訓
- **設計判断**: 背景 → 選択肢 → 決定 → 備考
- **学び / TIL**: ポイント

## フォーマット

- rawのmarkdownでの可読性を優先する
- 見出しは `#` と `##` を主に使う。`###` 以降は必要な場合のみ
- 太字ラベル（`**キー**: 値`）を多用しない。箇条書きで十分なら省略する
- コードスニペットには言語を指定する（`java`、`rust` 等）
- 日本語で出力する
- 会話内容を忠実に反映する。推測や補足で情報を追加しない
- ユーザーが繰り返し強調した内容はポイントの先頭に置くか、太字にする
