# 概要

複数リポジトリにまたがる特定トピック（API 使用箇所、設定キー、特定関数の呼び出し、非推奨 API の参照など）を調査し、TSV にまとめる。
grep を主軸とし、文脈依存で grep だけでは判断しづらいケースは Explore agent を補助的に使う。

出力は CWD からの相対パスで行う。CWD の選び方および出力ディレクトリ構造は `2-1` を参照。

## ローカル設定

スキル実行時は最初に以下のローカル設定ファイルを本スキルディレクトリから読み込む。

- `repos.local.md` — 対象リポジトリ一覧（パス、対象拡張子、テスト除外パターン）
- `keyword-mapping.local.md` — ドメイン用語・社内略称のキーワードマッピング

いずれかが存在しない場合は対応する `*.example.md` を参照してセットアップを案内する（コピーして編集を促す）。`repos.local.md` が存在しない、または `/path/to/...` のプレースホルダのままの場合はユーザーに実パスを確認する。

# 発動条件

ユーザーが `/codebase-scan` を明示的に呼び出した場合にのみ実行する。**自動発動はしない**（指示文中に「調査して」「使用箇所」「影響範囲」などの語が含まれていても、`/codebase-scan` の明示がない限り起動しない。それらの指示は単一リポジトリ内検索の可能性もあるため、Bash の grep や Explore agent で対応する）。

## 起動時の前提

- `repos.local.md` が存在し、対象リポジトリが 2 つ以上定義されている（未設定の場合は起動せず、ローカル設定セクションを参照してセットアップを案内する）
- 調査内容が複数リポジトリ横断であること（`/codebase-scan <調査内容>` のように指示が伴う想定）
- 調査が単一リポジトリで足りそうと判断した場合は、本スキルを使わずに済む方法をユーザーに提案する

# 手順

## 1. 調査設計（ユーザー承認が必須）

ユーザーの指示から以下を設計し、**箇条書きで提示してユーザーの承認を得る**。承認前に grep 等は一切実行しない。

### 1-1. 調査名

指示内容から短く命名する。（例: `認証処理の調査`, `非推奨API参照`）
出力ディレクトリ `./output/codebase-scan/<調査名>/<YYYYMMDD-HHMM>/` にそのまま使用するため、ファイルシステムで扱えない文字（`/`, `\`, `:`, `*`, `?`, `"`, `<`, `>`, `|`, 改行、先頭末尾空白）は避ける。空白はハイフンまたはアンダースコアに置き換える。

### 1-2. キーワード候補

本スキルの精度はこのステップで決まる。ユーザーの抽象的な指示（例:「認証を調査して」）を、grep で実行可能な具体語・正規表現に落とし込む作業。以下の 4 ステップで設計する。

**各ステップの責務**:

- **Step A. 調査意図の分類** — 「何を見つけるか」を言語化する
- **Step B. 候補文字列の列挙** — マッピング読み込み + 機械的発散。**数を気にせず広く取る**
- **Step C. ノイズ度の評価と絞り込み** — Step B の候補から最終投入分を選別する
- **Step D. grep 正規表現への組み立て** — Step C で残した候補を ERE パターンに統合する

#### Step A. 調査意図の分類

「何を見つけたいのか」を先に言語化する。意図によって有効なキーワード種別が変わるため、ここを曖昧にしたまま発散すると見つからない・ノイズが爆発する。

| 意図               | 典型的な指示                                   | 有効な狙い方                                          |
| ------------------ | ---------------------------------------------- | ----------------------------------------------------- |
| 宣言・定義箇所     | 「関数 X はどこで定義されているか」            | 宣言構文 + 識別子（`function X`, `class X`, `def X`） |
| 呼び出し・使用箇所 | 「X はどこから呼ばれているか」「X の影響範囲」 | 呼び出しパターン（`X(`, `.X(`, `import X`）           |
| 設定値・定数参照   | 「フラグ Y を参照している箇所」                | 定数名 + 文字列リテラル両方                           |
| アンチパターン検出 | 「`any` 型の残存」「生 SQL の使用」            | パターン正規表現 + 除外ファイル指定                   |
| 類似機能の発見     | 「認証処理のバリエーション」「似た実装の重複」 | 概念語・類義語・関連語の広いネット                    |

複数意図が混ざる場合は分けて設計する（同じ調査名内でキーワードをグループ化）。意図が明確でない場合は最も近い類型を仮選択し、承認ステップでユーザーに確認する。いずれの類型にも当てはまらない場合は「その他」として扱い、Step B での発散軸を絞り込まず広く取る。

#### Step B. キーワードを発散させる

意図を決めたら、以下の軸で候補を機械的に広げる。絞り込みは Step C で行う。

**最初に本スキルディレクトリの `keyword-mapping.local.md` を読み込む**（未設置の場合はローカル設定セクションを参照してセットアップを案内）。`日本語概念: 候補1, 候補2, ...` 形式のエントリで、指示内の日本語概念が左辺に一致するものがあれば、右辺の候補をすべてキーワードの種として**優先投入**する。マッピングはプロジェクト固有の独自略称を正確に反映しているため、機械生成より信頼度が高い。マッピング未記載の概念および残りの軸は、以下で機械的に展開する。

- **直接語**: API 名・関数名・クラス名・定数名・パッケージ名・URL パス・DB テーブル名・設定キー・エラーメッセージ・ログ文言
- **ケースの表記揺れ**: camelCase / snake_case / kebab-case / PascalCase / SCREAMING_SNAKE / 空白区切り（文字列リテラル内）
- **日本語概念 → 英数語化**（指示に日本語名詞が含まれる場合、以下をすべて機械的に展開する。日本のコードベースは命名が英訳・ローマ字・省略形で揺れるため網羅が必須）:
  - **英訳（複数候補）**: 多義な語は訳を絞らず全部出す
    - 例: 職員 → `employee` / `staff` / `worker` / `member`
    - 例: 情報 → `info` / `information` / `data`
    - 例: 担当者 → `person` / `owner` / `charge` / `assignee`
  - **ヘボン式ローマ字**: 職員 → `shokuin`、情報 → `jouhou`、人事 → `jinji`、申請 → `shinsei`
  - **訓令式ローマ字**: `syokuin` / `zyouhou` / `zinzi`（古い日本企業コードで混在）
  - **長音の表記ゆれ**: `jouhou` / `johou` / `joho` / `jyouhou`、`shuushoku` / `shushoku`
  - **ローマ字略称**: 職員 → `shkin` / `shk`、社員番号 → `shaban`、処理 → `shori` / `shr`
  - **英語略称**: `employee` → `emp`、`information` → `info`、`identification` → `id`、`number` → `no` / `num`、`management` → `mgmt`
  - **複合語の組み合わせ**: 職員ID → `shokuinId` / `shokuin_id` / `shkId` / `employeeId` / `empId` / `staffId`
  - **和英混在**: `userJoho` / `userJouhou` / `shokuinInfo` など、和製英語が残っているケース
- **別名（リネーム・愛称）**: 旧称 ↔ 新称のリネーム経緯、プロジェクト固有の愛称（例:「会員」を `mb`、「人事」を `hr`）、ログ/UI の和英表記
- **関連概念**: 上位概念（例: OAuth → 認証）、下位概念（例: 認証 → JWT / Session / Token）、類義語、反意語（アンチパターン調査時）
- **周辺語**: 宣言探索 → `function`, `class`, `def`, `const`, `export`。呼び出し探索 → `(`, `.`, `import`, `require`, `from`。メタ情報 → `@` デコレータ・アノテーション

grep は候補が実在しなくても 0 件で終わるだけなのでコスト増は軽微。プロジェクト固有の独自略称（社内慣習）でユーザーしか知らないものはユーザーに確認する。

##### Step B 補足: 影響範囲調査向けの追加軸

修正の波及先を網羅したい調査（Step A 類型「呼び出し・使用箇所」）では、上記に加えて以下を展開する。

- **API パス文字列**: クロスプロジェクト追跡用。Java の `@*Mapping("/api/v1/foo")` ↔ Vue の `axios.get('/api/v1/foo')`、他サービスの `@FeignClient` 経由参照。パスパラメータ部（`/users/{id}` の `{id}`）は正規表現でマスクして比較
- **DTO / Entity フィールド名**: getter/setter（`get<Field>(`, `set<Field>(`）と直接参照（`.<field>`）。Lombok 利用時はソースに getter/setter が無いことに注意
- **interface / 抽象クラス変更**: `implements\s+<Name>`、`extends\s+<Name>` で実装クラスを抽出
- **動的参照（要確認カテゴリ）**: 静的 grep だけでは断定できないが見落とすと事故になるもの。文字列ベース DI（`@Qualifier("name")`, `@Value("${...}")`, `getBean(String)`）、文字列キー dispatch（`dispatch('name')`, `commit('name')`）、リフレクション（`Class.forName`, `Method.invoke`）、動的 import / コンポーネント（`import(...)`, `<component :is="...">`）

#### Step C. 絞り込みとノイズ予測

発散したキーワードを以下の観点で篩にかけ、最終投入分を選別する。各候補にはノイズ度（低 / 中 / 高）を付記し、承認時の議論材料にする。

- **短すぎる一般語は強化 or 除外**: `get`, `set`, `id`, `handle`, `data` のような3-4文字の語は単独で使わない。単語境界・前後文脈で絞れなければ落とす
- **コメント/ドキュメントだけのヒットが多そうな語**: 設計書的な語は除外し、実装で使われている識別子に寄せる
- **テスト/モックだけに現れる語**: 除外方針は 1-4 のテスト除外オプションで対応
- **想定ノイズ度の判定基準**: 以下を目安に 低 / 中 / 高 を付記する。基準が曖昧な場合は一つ上のレベルを選ぶ（安全側）
  - **低**: 固有名詞・長い識別子（7 文字以上）・呼び出しパターン付き。ほぼ真のヒットのみ
  - **中**: 中程度の長さの識別子、または一般語だが前後文脈で絞っている。コメント / テストが混ざる可能性あり
  - **高**: 3-4 文字の一般語、単語境界のみで絞った語。代替案と併せて提示する

#### Step D. 正規表現化

grep `-E`（ERE）前提。macOS 標準 grep は BSD 系で `\b` が使えないため、以下のイディオムで書く。

- 単語境界（BSD 互換）: `(^|[^A-Za-z0-9_])keyword([^A-Za-z0-9_]|$)`
- 表記揺れまとめ: `(foo_bar|fooBar|FooBar)`
- 関数呼び出し限定: `\.?keyword\(`
- インポート限定: `(import|require|from)[^\n]*keyword`
- ファイル除外は正規表現ではなく grep の `--exclude-dir`, `--exclude` で指定する（ERE は否定先読み非対応）

### 1-3. 対象リポジトリ

全リポジトリ対象がデフォルト。ユーザからリポジトリの指示があるか、指示内容から絞り込みが明らかに妥当なら候補を絞って提案する（例: フロント固有のキーワードならフロントのみ）。

### 1-4. 除外パターン

検索から除外するディレクトリ・ファイル種別を列挙する。既定:

- `node_modules`, `vendor`, `dist`, `build`, `target`, `.git`, `.next`, `.nuxt`, `coverage`
- ロックファイル（`*-lock.json`, `*.lock`）、ミニファイ済みファイル（`*.min.*`）
- 明らかに調査対象外のバイナリ・画像・フォント

対象拡張子を絞るべきかも提案する（例: TypeScript のみなら `*.ts,*.tsx`）。

**テスト除外**: 承認フェーズで「テスト除外 する／しない」を選ぶ。除外する場合、`repos.local.md` の対象リポジトリテーブル「テスト除外パターン」列に定義された値を `--exclude-dir` / `--exclude` に追加する。リポジトリごとにパターンが異なる想定なので、grep コマンドは**リポジトリ別に組み立てる**。

### 1-5. 提示フォーマット

上記を以下のように箇条書きで提示し、「この内容で実行してよいか」確認する。ユーザー修正に応じて再提示する。

```
【調査設計案】
- 調査名: 職員ID参照箇所
- 調査意図: 呼び出し・使用箇所の洗い出し
- 対象リポジトリ: a, b, c
- キーワード:
  - (shokuinId|shokuin_id|employeeId|empId|staffId|shkId)  狙い: 職員ID フィールド全表記         ノイズ: 低
  - (userInfo|userJoho|userJouhou|user_info|userData)      狙い: ユーザー情報オブジェクトの参照   ノイズ: 中
  - (syokuinId|zyouhou)                                     狙い: 訓令式ローマ字の残存             ノイズ: 低
  - "職員ID"|"ユーザー情報"|"ユーザ情報"                    狙い: ログ/UI 文字列・コメントの検出   ノイズ: 高 ← コメントも拾う
- 除外: node_modules, dist, *.min.*
- 対象拡張子: *.ts, *.tsx, *.java
- テスト除外: する／しない

この内容で実行してよいですか？
```

## 2. 並列 grep 実行

承認を得たら、**対象リポジトリごとに Bash tool call を単一メッセージ内で並列発行**する（マルチプロセス実行）。bash のバックグラウンドジョブ（`&`）は使わない（ログ取り逃しを防ぐため）。リポジトリがボリューム大のため、順次実行はしない。

実行環境の前提は「注意事項」を参照。

### 2-1. 出力先の準備

出力は CWD 配下の `./output/codebase-scan/<調査名>/<YYYYMMDD-HHMM>/` に作成する。**CWD は対象リポジトリの外部に置くこと**（リポジトリ内に出力すると `git status` が汚染される）。

ディレクトリ構造:

```
<CWD>/output/codebase-scan/<調査名>/<YYYYMMDD-HHMM>/
├── scan-result.tsv      # 最終統合結果
├── notes.md             # Agent 深堀り結果（4章実施時のみ作成）
└── .tmp/
    ├── scan-design.md   # 承認された調査設計（人が読む形式）
    ├── <repo1>.raw      # 全キーワードの grep 結果（行ごとに <repo>\t<keyword>\t プレフィックス付き）
    ├── <repo2>.raw
    └── <repo3>.raw
```

`YYYYMMDD-HHMM` は `date +%Y%m%d-%H%M` で取得する。同一 `<調査名>` の履歴が同じディレクトリ（`./output/codebase-scan/<調査名>/`）配下に時系列で並ぶ。

### 2-2. 調査設計の保存

grep 実行前に、承認された調査設計を `.tmp/scan-design.md` に書き出す。内容は `1-5` の提示フォーマット（ユーザー修正を反映した最終版）をそのまま保存する。用途:

- **再現性**: 後日同じ調査を再実行する際のパラメータ記録
- **監査**: 第三者が調査意図・キーワード選定を確認できる

### 2-3. grep コマンド

**キーワードごとに個別に grep を実行**する（OR 連結はしない）。出力に `<repo_name>\t<keyword_name>\t` プレフィックスを付けて連結し、`.tmp/<repo>.raw` にリダイレクトする。これにより 3 段階目の TSV 統合時に、どのキーワードが当たったかが grep 出力時点で確定する（再マッチング処理が不要）。

リポジトリ単位で 1 つの Bash tool call にまとめ、リポジトリ間は単一メッセージ内で並列発行する。リポジトリ内の各キーワード grep は順次実行（OS のファイルキャッシュにより 2 回目以降は高速）。

構成規則:

- 固定: `grep -rnIE` フラグ
  - `-r` 再帰、`-n` 行番号、`-I` バイナリ除外、`-E` ERE
- 可変（1-4 / 1-5 で確定した内容に置き換え）:
  - `--include` — 1-4 で対象拡張子を絞った場合のみ指定。省略可
  - `--exclude-dir` — 1-4 既定 + リポジトリ別テスト除外（`repos.local.md` のテーブル）
  - `--exclude` — 1-4 既定（ロックファイル、ミニファイ等）
  - 検索パターン — 1-2 で確定したキーワードを**1つずつ**個別に渡す
  - 検索パス — `cd <repo_path> &&` 後に `.` を渡す（出力ファイルパスを相対化するため）
- プレフィックス付与: 各 grep の出力を `awk -v p='<repo>\t<keyword_name>\t' '{print p $0}'` でラップ
- 出力先: `cd <repo_path> && { ...all-keyword-greps... } > <work_dir>/.tmp/<repo>.raw`
- パイプで `| head -n 10000` 等は**かけない**。全件取得する。過大な場合は次ステップで警告する
- 終了コード 1（マッチなし）は想定内なのでエラーとして扱わない

### 2-4. 件数確認

`.raw` 全体の行数だけでなく、**キーワードごと**のヒット数を集計してノイズの偏りを検出する:

```bash
# 全体行数
wc -l <work_dir>/.tmp/*.raw

# キーワード別ヒット数（リポジトリ別）
for f in <work_dir>/.tmp/*.raw; do
  echo "== $(basename "$f" .raw) =="
  awk -F'\t' '{print $2}' "$f" | sort | uniq -c | sort -rn
done
```

以下のいずれかに該当する場合はユーザーに報告して判断を仰ぐ:

- 1 キーワードあたり概ね 1000 行超のヒット — そのキーワードはノイズが多すぎる可能性。絞り込みを提案する
- 他のキーワード・他リポジトリと比較して件数が大きく外れている — 設計の見直しが妥当
- ヒット 0 のキーワード — 表記漏れや別経路の可能性。Agent 深堀り（4章）の候補

（「1000 行」は分析に耐える件数の目安。プロジェクト規模により増減してよい）

## 3. 結果の統合

各 `.raw` を読み込み、`./output/codebase-scan/<調査名>/<YYYYMMDD-HHMM>/scan-result.tsv` に統合する。

### 3-1. TSV 構造

ヘッダ付き。タブ区切り。

```
repo	file	line	keyword	match_text
```

| 列         | 内容                                                          |
| ---------- | ------------------------------------------------------------- |
| repo       | 対象リポジトリ名（`repos.local.md` の `名前` 列）             |
| file       | リポジトリルートからの相対パス                                |
| line       | 行番号（整数）                                                |
| keyword    | マッチしたキーワード。複数マッチする行は `,` 区切りで複数列挙 |
| match_text | マッチ行の内容。タブは半角空白に置換。先頭末尾の空白は除去    |

### 3-2. 整形ルール

- `file` はリポジトリのルートパスを前方一致で削除して相対化する
- `match_text` 内のタブ・改行は半角空白に置換する（TSV 壊れ防止）
- `match_text` が 500 文字を超える場合は 500 文字で切り詰め、末尾に `…` を付与
- ソート順: `repo` → `file` → `line` の昇順

### 3-3. TSV への統合

`.raw` の各行はすでに `<repo>\t<keyword>\t<file>:<line>:<text>` 形式で keyword が確定しているため、再マッチングは不要。`.tmp/*.raw` をマージして `scan-result.tsv` を生成する。

**実装は同梱の `scripts/build-tsv.sh` を使う**:

```bash
bash <スキルパス>/scripts/build-tsv.sh ./output/codebase-scan/<調査名>/<YYYYMMDD-HHMM>/
```

スクリプトの挙動:

1. `.tmp/*.raw` を結合し、各行を `<file>:<line>:<text>` 部分でパースして 3-2 の整形ルールを適用
2. `(repo, file, line, keyword)` 昇順でソート
3. 連続する同 `(repo, file, line)` の `keyword` を `,` 区切りで連結（出現順）
4. ヘッダ付き TSV を `scan-result.tsv` に書き出す（UTF-8 / LF 改行 / BOM なし）

`.tmp/*.raw` が存在しない、もしくは全て空（全リポジトリで 0 件）の場合は、ヘッダ行のみの TSV を生成して 0 件として報告する。awk + sort の標準コマンドのみで動作するため、追加ランタイム（Python 等）は不要。

### 3-4. 後片付け

`.tmp/` は調査のトレースとして残す（削除しない）。容量が気になる場合はユーザーから指示があれば削除する。

## 4. Agent による深堀り調査（任意・ユーザー承認）

grep 結果を評価したうえで、以下のケースに該当する場合は **Claude からユーザーに「Agent で深堀りするか」を提案し、承認を得てから** Explore agent をリポジトリ単位で並列起動する。自動では起動しない（コンテキスト・時間コストが大きいため）。

- キーワードが一般語で grep ではノイズが多すぎる（例: `handle`, `get`, `data`）
- 呼び出し関係や継承関係など「意味的な」追跡が必要
- マッチ件数 0 だが、別名・別経路で同等機能が実装されている可能性がある

結果は `./output/codebase-scan/<調査名>/<YYYYMMDD-HHMM>/notes.md` に追記する。scan-result.tsv 側には追加しない（TSV は grep の生データに限定する）。

Agent には以下を明示的に伝える:

- 対象リポジトリパス
- 調査目的（ユーザー指示の原文）
- 既に grep で得たヒット（`.raw` のパス）
- 求める成果（ノイズ判定、関連箇所、別名候補など）
- 文字数制限（report under 400 words 等）

## 5. 最終報告

ユーザーへの報告は以下をコンパクトに伝える。scan-result.tsv の中身を長々と貼らない。

- 出力ファイル: `./output/codebase-scan/<調査名>/<YYYYMMDD-HHMM>/scan-result.tsv` の絶対パス
- 総ヒット件数
- リポジトリ別ヒット件数
- キーワード別ヒット件数（上位のみ）
- 特筆事項（想定外のマッチ、0 件キーワード、ノイズが多いキーワード、補助 `notes.md` の有無）

**総ヒット 0 件の場合**: scan-result.tsv はヘッダのみになる。キーワード設計の見直し（表記揺れの追加、別経路の探索）または Agent 深堀り（4章）の利用を提案する。

# 注意事項

- 動作環境: bash + 標準 coreutils（grep, awk, sort, wc）が必須
  - macOS / Linux / WSL: 標準で動作
  - Windows: **Git for Windows 同梱の Git Bash を使用する**（または WSL）。VS Code 等で本スキルを呼び出す場合は、統合ターミナルのデフォルトプロファイルを Git Bash に設定しておく
- 日本語キーワードを含める場合は `LC_ALL=C.UTF-8` 環境を前提にする（macOS / Linux / Git Bash いずれも標準で問題ない）
- ファイル文字コードは UTF-8 を前提にする。Shift-JIS や CP932 のソースを扱う場合は事前に変換する
- 本スキルはリードオンリー。コードを書き換えない
