# Skills vs Agents 設計指針

## 概要

Claude Code における Skill と Agent の使い分けの設計方針を定める。
DDD / クリーンアーキテクチャの Service / UseCase パターンを参考にする。

## 対応関係

| DDD / クリーンアーキ    | Claude Code         | 役割                                         |
| ------------------------ | ------------------- | -------------------------------------------- |
| Service                  | Skill               | 単一責務の操作を提供する                     |
| UseCase / Interactor     | Agent               | 複数の Skill を組み合わせて目的を達成する    |
| Controller               | ユーザー or 親Agent | どの UseCase を呼ぶか判断する                |

## Skill（= Service）

- **単一責務**に集中する
- 使えるツールを最小限に絞る（`allowed-tools`）
- 自分の範囲外の処理をしない
- `context: fork` で独立コンテキストを活用できる

### 設計基準

- 1つのスキルが1つの動詞で説明できること（レビューする、記録する、抽出する）
- 他のスキルを呼ばない（フラットな構造）
- 読み取り専用のスキルには書き込みツールを与えない

### 例

```
review-code  → コード変更をレビューする（読み取り専用）
note         → 知見をmarkdownとして保存する
triage-errors → ログからスタックトレースを抽出する
```

## Agent（= UseCase）

- 複数のステップを**オーケストレーション**する
- Skill や Claude 本体の標準能力を組み合わせてワークフローを実現する
- ドメイン固有の制約がある操作は Skill に委譲する
- 汎用的な操作（コード修正、ファイル作成など）は Agent 自身が直接行ってよい

### 設計基準

- 2つ以上のステップを自律的に実行する必要があること
- 単一のスキルで完結するなら Agent にしない
- ワークフロー全体の制御フローを知っている
- 全てを Skill 経由にする必要はない — Skill にすべきかの判断は前述の基準に従う

### 例

```
auto-review-fix:
  1. review-code スキル（または同等のロジック）でレビュー
  2. 指摘を元にコードを修正
  3. 再度レビューして確認
```

## 判断フローチャート

```
やりたいことは1つの動詞で表現できる？
  ├─ Yes → Skill
  └─ No → 複数ステップの自律的な実行が必要？
            ├─ Yes → Agent
            └─ No → ユーザーが手動で Skill を順に呼ぶ
```

## アンチパターン

### Skill に複数の責務を持たせる

```
# NG: review-and-fix スキル
1. レビューする
2. 修正する        ← レビューアーが自分で直すと客観性が失われる
3. 再レビューする  ← 自分の修正を自分でレビューする意味がない
```

**問題点:**
- レビューの客観性が失われる
- `allowed-tools` に書き込み権限が必要になり、安全性の境界が曖昧になる
- プロンプトが肥大化し、各ステップの品質が下がる

### Agent を使わず全てを1つの Skill に詰め込む

Skill のプロンプトが長くなり、条件分岐が増えてきたら Agent への分離を検討する。

## 今後の展開

必要に応じて以下のような Agent を追加する：

| Agent             | 概要                                             |
| ----------------- | ------------------------------------------------ |
| `auto-review-fix` | レビュー → 修正 → 再レビューの自動化             |
| `log-investigate`  | エラー抽出 → 原因調査 → 修正提案                |
